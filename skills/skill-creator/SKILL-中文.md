# 技能创建器

一个用于创建新技能并迭代改进它们的技能。

在高层次上，创建技能的过程如下：

- 决定您希望技能做什么以及大致如何做
- 起草技能
- 创建一些测试提示并在其上运行具有技能访问权限的claude
- 帮助用户定性和定量地评估结果
  - 当运行在后台进行时，如果没有评估，起草一些定量评估（如果有，您可以按原样使用，或者如果觉得需要更改某些内容则进行修改）。然后向用户解释它们（或者如果它们已经存在，则解释已经存在的那些）
  - 使用 `eval-viewer/generate_review.py` 脚本向用户显示结果供他们查看，并让他们查看定量指标
- 根据用户评估结果的反馈重写技能（以及如果定量基准测试中出现了任何明显缺陷）
- 重复直到满意
- 扩大测试集并再次尝试更大规模

使用此技能时，您的工作是弄清楚用户在此过程中的位置，然后跳进去帮助他们完成这些阶段。例如，也许他们会说"我想为X制作一个技能"。您可以帮助缩小他们的意思，起草，编写测试用例，弄清楚他们想要如何评估，运行所有提示，并重复。

另一方面，也许他们已经起草了技能。在这种情况下，您可以直接进入循环的评估/迭代部分。

当然，您应该始终保持灵活，如果用户说"我不需要运行大量评估，只是跟我一起感受"，您可以这样做。

然后在技能完成后（但同样，顺序是灵活的），您还可以运行技能描述优化器，我们有一个单独的脚本，以优化技能的触发。

酷吗？酷。

## 与用户沟通

技能创建器可能会被各种熟悉编码术语的人使用。如果您还没有听说过（您怎么可能听说，它才刚刚开始），现在有一个趋势，Claude的力量激励水管工打开他们的终端，父母和祖父母谷歌"如何安装npm"。另一方面，大多数用户可能相当精通计算机。

所以请注意上下文线索，了解如何表达您的沟通！在默认情况下，只是为了给您一些想法：

- "评估"和"基准"是边缘的，但可以接受
- 对于"JSON"和"断言"，您希望看到用户认真提示他们知道这些是什么，然后才在不解释它们的情况下使用它们

如果您有疑问，简要解释术语是可以的，如果您不确定用户是否会理解，请随时用简短定义澄清术语。

## 创建技能

### 捕获意图

首先了解用户的意图。当前对话可能已经包含用户想要捕获的工作流程（例如，他们说"将其转换为技能"）。如果是这样，首先从对话历史中提取答案 - 使用的工具、步骤序列、用户所做的更正、观察到的输入/输出格式。用户可能需要填补空白，并应在继续下一步之前确认。

1. 这个技能应该使Claude能够做什么？
2. 这个技能何时应该触发？（什么用户短语/上下文）
3. 预期的输出格式是什么？
4. 我们应该设置测试用例来验证技能是否有效？具有客观可验证输出的技能（文件转换、数据提取、代码生成、固定工作流程步骤）受益于测试用例。具有主观输出的技能（写作风格、艺术）通常不需要它们。根据技能类型建议适当的默认值，但让用户决定。

### 访谈和研究

主动询问边缘情况、输入/输出格式、示例文件、成功标准和依赖关系。在弄清楚这部分之前，先不要编写测试提示。

检查可用的MCP - 如果有助于研究（搜索文档、查找类似技能、查找最佳实践），如果可用，通过子代理并行研究，否则内联。带着背景信息来，以减少用户的负担。

### 编写SKILL.md

根据用户访谈，填写这些组件：

- **name**: 技能标识符
- **description**: 何时触发，做什么。这是主要的触发机制 - 包括技能做什么以及何时使用的具体上下文。所有"何时使用"信息都在这里，而不是正文中。注意：目前Claude倾向于"欠触发"技能 - 在有用时不使用它们。为了解决这个问题，请让技能描述有点"主动"。例如，而不是"如何构建一个简单的快速仪表板来显示内部Anthropic数据。"，您可能会写"如何构建一个简单的快速仪表板来显示内部Anthropic数据。确保每当用户提到仪表板、数据可视化、内部指标或想要显示任何类型的公司数据时，即使他们没有明确要求'仪表板'，也要使用此技能。"
- **compatibility**: 必需的工具、依赖关系（可选，很少需要）
- **技能的其余部分 :)**

### 技能写作指南

#### 技能的解剖结构

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML前言（name、description必需）
│   └── Markdown指令
└── 捆绑资源（可选）
    ├── scripts/    - 用于确定性/重复性任务的可执行代码
    ├── references/ - 根据需要加载到上下文中的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进式披露

技能使用三级加载系统：
1. **元数据**（name + description）- 始终在上下文中（约100字）
2. **SKILL.md正文** - 技能触发时在上下文中（理想情况下少于500行）
3. **捆绑资源** - 根据需要（无限，脚本可以在不加载的情况下执行）

这些字数统计是近似的，如果需要，您可以随意延长。

**关键模式：**
- 保持SKILL.md在500行以下；如果您接近这个限制，添加额外的层次结构以及关于使用此技能的模型下一步应该去哪里的明确指针。
- 从SKILL.md清楚地引用文件，并提供何时阅读它们的指导
- 对于大型参考文件（>300行），包括目录

**域组织**：当技能支持多个域/框架时，按变体组织：
```
cloud-deploy/
├── SKILL.md (工作流程 + 选择)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude只读取相关的参考文件。

#### 缺乏惊喜原则

这不用说，但技能不得包含恶意软件、利用代码或任何可能危害系统安全的内容。技能的内容不应在描述时让用户感到惊讶。不要配合创建误导性技能或旨在促进未经授权访问、数据泄露或其他恶意活动的技能。像"扮演XYZ"这样的事情是可以的。

#### 写作模式

在指令中优先使用祈使语气。

**定义输出格式** - 您可以这样做：
```markdown
## 报告结构
始终使用此确切模板：
# [标题]
## 执行摘要
## 关键发现
## 建议
```

**示例模式** - 包括示例很有用。您可以像这样格式化它们（但如果示例中有"输入"和"输出"，您可能想要稍微偏离）：
```markdown
## 提交消息格式
**示例1：**
输入：添加了使用JWT令牌的用户身份验证
输出：feat(auth): 实现基于JWT的身份验证
```

### 写作风格

尝试向模型解释为什么事情很重要，而不是沉重的必须MUSTs。使用心智理论，尝试使技能通用而不是超级狭窄的特定示例。先写草稿，然后用新的眼光看它并改进它。

### 测试用例

写完技能草稿后，提出2-3个现实的测试提示 - 真实用户实际会说的那种。与用户分享：[您不必使用这个确切语言]"这里有几个我想尝试的测试用例。这些看起来对吗，或者您想添加更多？"然后运行它们。

将测试用例保存到 `evals/evals.json`。还没有写断言 - 只是提示。您将在下一步中起草断言，而运行正在进行中。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示",
      "expected_output": "预期结果描述",
      "files": []
    }
  ]
}
```

有关完整模式（包括您将稍后添加的 `assertions` 字段），请参见 `references/schemas.md`。

## 运行和评估测试用例

本节是一个连续的序列 - 不要中途停止。不要使用 `/skill-test` 或任何其他测试技能。

将结果放在 `<skill-name>-workspace/` 中作为技能目录的兄弟目录。在工作区内，按迭代（`iteration-1/`、`iteration-2/` 等）组织结果，在其中，每个测试用例获得一个目录（`eval-0/`、`eval-1/` 等）。不要预先创建所有这些 - 只是在进行时创建目录。

### 步骤1：在同一轮中生成所有运行（有技能和无技能）

对于每个测试用例，在同一轮中生成两个子代理 - 一个有技能，一个没有。这很重要：不要先生成有技能的运行，然后再回来生成基线。一次性启动所有内容，以便它们大约在同一时间完成。

**有技能运行：**

```
执行此任务：
- 技能路径：<技能路径>
- 任务：<评估提示>
- 输入文件：<评估文件（如果有）或"无">
- 保存输出到：<工作区>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 要保存的输出：<用户关心的内容 - 例如，".docx文件"、"最终CSV">
```

**基线运行**（相同的提示，但基线取决于上下文）：
- **创建新技能**：根本没有技能。相同的提示，没有技能路径，保存到 `without_skill/outputs/`。
- **改进现有技能**：旧版本。在编辑之前，快照技能（`cp -r <技能路径> <工作区>/skill-snapshot/`），然后将基线子代理指向快照。保存到 `old_skill/outputs/`。

为每个测试用例写一个 `eval_metadata.json`（断言现在可以为空）。给每个评估一个基于它所测试内容的描述性名称 - 而不仅仅是"eval-0"。也用这个名称作为目录名称。如果此迭代使用新的或修改的评估提示，为每个新评估目录创建这些文件 - 不要假设它们从以前的迭代中延续过来。

```json
{
  "eval_id": 0,
  "eval_name": "这里描述性名称",
  "prompt": "用户的任务提示",
  "assertions": []
}
```

### 步骤2：在运行进行时，起草断言

不要只是等待运行完成 - 您可以有效地利用这段时间。为每个测试用例起草定量断言并向用户解释它们。如果 `evals/evals.json` 中已经存在断言，请查看它们并解释它们检查的内容。

好的断言是客观可验证的，并且具有描述性名称 - 它们应该在基准查看器中清晰地阅读，以便有人瞥一眼结果就能立即理解每个检查的内容。主观技能（写作风格、设计质量）最好定性评估 - 不要强迫断言到需要人类判断的事物上。

起草后更新 `eval_metadata.json` 文件和 `evals/evals.json` 中的断言。还要向用户解释他们将在查看器中看到什么 - 包括定性输出和定量基准。

### 步骤3：当运行完成时，捕获时间数据

当每个子代理任务完成时，您会收到包含 `total_tokens` 和 `duration_ms` 的通知。立即将此数据保存到运行目录中的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会 - 它通过任务通知传来，不会在其他地方持久化。在通知到达时处理每个通知，而不是尝试批量处理它们。

### 步骤4：评分、汇总并启动查看器

一旦所有运行完成：

1. **对每个运行进行评分** - 生成一个评分子代理（或内联评分），读取 `agents/grader.md` 并评估每个断言与输出。将结果保存到每个运行目录中的 `grading.json`。grading.json期望数组必须使用字段 `text`、`passed` 和 `evidence`（而不是 `name`/`met`/`details` 或其他变体）- 查看器依赖于这些确切的字段名称。对于可以编程检查的断言，编写并运行脚本而不是用眼睛看 - 脚本更快、更可靠，并且可以在迭代中重复使用。

2. **汇总到基准** - 从技能创建器目录运行汇总脚本：
   ```bash
   python -m scripts.aggregate_benchmark <工作区>/iteration-N --skill-name <名称>
   ```
   这将生成 `benchmark.json` 和 `benchmark.md`，其中包含每种配置的通过率、时间和令牌，具有平均值±标准差和增量。如果手动生成benchmark.json，请参见 `references/schemas.md` 了解查看器期望的确切模式。
在每个基线对应项之前放置每个有技能的版本。

3. **进行分析通过** - 读取基准数据并显示汇总统计可能隐藏的模式。参见 `agents/analyzer.md`（"分析基准结果"部分）了解要查找的内容 - 像无论技能如何都通过的断言（非歧视性）、高方差评估（可能不稳定）以及时间/令牌权衡这样的事情。

4. **启动查看器**，同时显示定性输出和定量数据：
   ```bash
   nohup python <技能创建器路径>/eval-viewer/generate_review.py \
     <工作区>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <工作区>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   对于迭代2+，还要传递 `--previous-workspace <工作区>/iteration-<N-1>`。

   **Cowork/无头环境：** 如果 `webbrowser.open()` 不可用或环境没有显示，请使用 `--static <output_path>` 写入独立的HTML文件而不是启动服务器。当用户点击"提交所有评论"时，反馈将作为 `feedback.json` 文件下载。下载后，将 `feedback.json` 复制到工作区目录中，以便下次迭代获取。

注意：请使用generate_review.py创建查看器；无需编写自定义HTML。

5. **告诉用户** 类似的话："我已在浏览器中打开结果。有两个选项卡 - '输出'让您可以点击每个测试用例并留下反馈，'基准'显示定量比较。完成后，回到这里告诉我。"

### 用户在查看器中看到的内容

"输出"选项卡一次显示一个测试用例：
- **提示**：给定的任务
- **输出**：技能产生的文件，尽可能内联呈现
- **以前的输出**（迭代2+）：折叠部分显示上次迭代的输出
- **正式评分**（如果运行了评分）：折叠部分显示断言通过/失败
- **反馈**：随他们输入自动保存的文本框
- **以前的反馈**（迭代2+）：他们上次评论，显示在文本框下方

"基准"选项卡显示统计摘要：每种配置的通过率、时间和令牌使用量，具有每次评估的细分和分析师观察。

导航通过上一个/下一个按钮或箭头键进行。完成后，他们点击"提交所有评论"，将所有反馈保存到 `feedback.json`。

### 步骤5：阅读反馈

当用户告诉您他们完成后，阅读 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完美，喜欢这个", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈意味着用户认为没问题。专注于用户有具体投诉的测试用例的改进。

完成后杀死查看器服务器：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是循环的核心。您已经运行了测试用例，用户已经审查了结果，现在您需要基于他们的反馈使技能更好。

### 如何思考改进

1. **从反馈中泛化。** 这里发生的大局事情是我们试图创建可以（也许是字面上的，甚至更多谁知道）使用数百万次的技能，跨越许多不同的提示。在这里，您和用户一遍又一遍地迭代仅几个示例，因为它有助于更快地移动。用户对这些示例了如指掌，他们快速评估新输出很容易。但如果您和用户共同开发的技能仅适用于这些示例，它就毫无用处。而不是放入繁琐的过度拟合更改或压迫性的限制性MUSTs，如果有一些顽固的问题，您可能会尝试分支并使用不同的隐喻，或推荐不同的工作模式。尝试相对便宜，也许您会找到很棒的东西。

2. **保持提示精简。** 删除没有发挥作用的内容。确保阅读成绩单，而不仅仅是最终输出 - 如果看起来技能使模型浪费大量时间做没有成效的事情，您可以尝试删除使其这样做的技能部分，看看会发生什么。

3. **解释原因。** 努力解释您要求模型做的每件事的**原因**。今天的LLM是*聪明的*。他们有很好的心智理论，当给予良好的指导时，可以超越死记硬背，真正使事情发生。即使用户的反馈是简短的或沮丧的，也要尝试实际理解任务以及用户为什么写他们写的内容，他们实际写了什么，然后将这种理解转化为指令。如果您发现自己用大写字母写ALWAYS或NEVER，或使用超级严格的结构，那是一个黄色标志 - 如果可能，重新构建并解释推理，以便模型理解您要求的事情为什么重要。这是更人性化、更强大和更有效的方法。

4. **寻找测试用例中的重复工作。** 阅读测试运行的成绩单，注意子代理是否都独立编写了类似的助手脚本或对某事采取了相同的多步骤方法。如果所有3个测试用例都导致子代理编写 `create_docx.py` 或 `build_chart.py`，这是一个强烈的信号，技能应该捆绑该脚本。编写一次，放入 `scripts/`，并告诉技能使用它。这节省了每次未来的调用免于重新发明轮子。

这项任务非常重要（我们试图在这里创造每年数十亿的经济价值！），您的思考时间不是瓶颈；慢慢来，真正仔细考虑。我建议写修订草稿，然后用新的眼光看它并进行改进。真正尽力进入用户的头脑，了解他们想要什么和需要什么。

### 迭代循环

改进技能后：

1. 将您的改进应用到技能
2. 将所有测试用例重新运行到新的 `iteration-<N+1>/` 目录中，包括基线运行。如果您正在创建新技能，基线始终是 `without_skill`（没有技能）- 它在迭代中保持不变。如果您正在改进现有技能，请使用您的判断来确定什么作为基线有意义：用户进入时的原始版本，还是以前的迭代。
3. 启动带有 `--previous-workspace` 指向以前迭代的审查器
4. 等待用户审查并告诉您他们完成了
5. 阅读新的反馈，再次改进，重复

继续直到：
- 用户说他们满意
- 反馈都是空的（一切看起来都不错）
- 您没有取得有意义的进展

## 高级：盲比较

对于您想要对技能的两个版本进行更严格比较的情况（例如，用户问"新版本实际上更好吗？"），有一个盲比较系统。阅读 `agents/comparator.md` 和 `agents/analyzer.md` 了解详细信息。基本想法是：给独立代理两个输出，不告诉它哪个是哪个，让它判断质量。然后分析获胜者获胜的原因。

这是可选的，需要子代理，大多数用户不需要。人工审查循环通常就足够了。

## 描述优化

SKILL.md前言中的描述字段是决定Claude是否调用技能的主要机制。创建或改进技能后，主动优化描述以获得更好的触发准确性。

### 步骤1：生成触发评估查询

创建20个评估查询 - 混合应该触发和不应该触发的查询。保存为JSON：

```json
[
  {"query": "用户提示", "should_trigger": true},
  {"query": "另一个提示", "should_trigger": false}
]
```

查询必须是现实的，并且是Claude Code或Claude.ai用户实际会输入的内容。不是抽象的请求，而是具体的、详细的、有良好上下文的请求。例如，文件路径、关于用户工作或情况的个人背景、列名和值、公司名称、URL。一点点背景故事。有些可能是小写的或包含缩写、拼写错误或随意讲话。使用不同长度的混合，专注于边缘情况而不是让它们明确（用户将有机会签署它们）。

不好：`"格式化这些数据"`、`"从PDF中提取文本"`、`"创建图表"`

好：`"好的，我的老板刚刚发送给我这个xlsx文件（它在下载中，名为类似'Q4销售最终最终v2.xlsx'），她希望我添加一列显示利润率百分比。收入在C列，成本在D列，我想"`

对于 **应该触发** 查询（8-10个），考虑覆盖范围。您想要相同意图的不同措辞 - 有些正式，有些随意。包括用户没有明确命名技能或文件类型但明确需要它的情况。加入一些不常见的用例和此技能与其他技能竞争但应该获胜的情况。

对于 **不应该触发** 查询（8-10个），最有价值的是近似错过 - 与技能共享关键词或概念但实际上需要不同内容的查询。考虑相邻域、模糊措辞，其中天真关键字匹配会触发但不应该触发，以及查询触及技能所做的某些内容但在另一个工具更合适的上下文中的情况。

关键要避免：不要使不应该触发的查询明显不相关。"编写斐波那契函数"作为PDF技能的负面测试太容易 - 它什么也不测试。负面情况应该真正棘手。

### 步骤2：与用户审查

使用HTML模板向用户展示评估集以供审查：

1. 从 `assets/eval_review.html` 读取模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → JSON评估项数组（周围没有引号 - 它是一个JS变量赋值）
   - `__SKILL_NAME_PLACEHOLDER__` → 技能的名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 技能的当前描述
3. 写入临时文件（例如 `/tmp/eval_review_<skill-name>.html`）并打开它：`open /tmp/eval_review_<skill-name>.html`
4. 用户可以编辑查询、切换应该触发、添加/删除条目，然后点击"导出评估集"
5. 文件下载到 `~/Downloads/eval_set.json` - 检查下载文件夹中的最新版本，以防有多个（例如 `eval_set (1).json`）

这一步很重要 - 糟糕的评估查询导致糟糕的描述。

### 步骤3：运行优化循环

告诉用户："这需要一些时间 - 我将在后台运行优化循环并定期检查它。"

将评估集保存到工作区，然后在后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <路径到触发评估.json> \
  --skill-path <路径到技能> \
  --model <模型ID驱动此会话> \
  --max-iterations 5 \
  --verbose
```

使用系统提示中的模型ID（驱动当前会话的那个），以便触发测试与用户实际体验的内容匹配。

运行时，定期查看输出，向用户更新它所在的迭代以及分数的样子。

这会自动处理完整的优化循环。它将评估集拆分为60%训练和40%保留测试，评估当前描述（运行每个查询3次以获得可靠的触发率），然后调用Claude根据失败的内容提出改进建议。它在训练和测试上重新评估每个新描述，最多迭代5次。完成后，它在浏览器中打开一个HTML报告，显示每次迭代的结果，并返回带有 `best_description` 的JSON - 选择通过测试分数而不是训练分数来选择，以避免过拟合。

### 技能触发如何工作

了解触发机制有助于设计更好的评估查询。技能出现在Claude的 `available_skills` 列表中，带有它们的名称+描述，Claude根据该描述决定是否咨询技能。重要的是要知道Claude只咨询它不能轻松处理自己的任务 - 简单的、一步的查询，如"阅读此PDF"可能不会触发技能，即使描述完美匹配，因为Claude可以直接使用基本工具处理它们。复杂的、多步的或专业的查询在描述匹配时可靠地触发技能。

这意味着您的评估查询应该足够实质性，以至于Claude实际上会从咨询技能中受益。像"读取文件X"这样的简单查询是很差的测试用例 - 无论描述质量如何，它们都不会触发技能。

### 步骤4：应用结果

从JSON输出中获取 `best_description` 并更新技能的SKILL.md前言。向用户显示前后对比并报告分数。

检查您是否可以访问 `present_files` 工具。如果不能，跳过这一步。如果可以，打包技能并向用户呈现.skill文件：

```bash
python -m scripts.package_skill <路径到技能文件夹>
```

打包后，将用户指向生成的 `.skill` 文件路径，以便他们可以安装它。

## Claude.ai特定说明

在Claude.ai中，核心工作流程相同（起草 → 测试 → 审查 → 改进 → 重复），但由于Claude.ai没有子代理，一些机制会改变。以下是要适应的内容：

**运行测试用例**：没有子代理意味着没有并行执行。对于每个测试用例，阅读技能的SKILL.md，然后按照其指示自己完成测试提示。一次做一个。这不如独立子代理严格（您编写了技能，您也在运行它，所以您有完整的上下文），但它是一个有用的健全性检查 - 人工审查步骤弥补了这一点。跳过基线运行 - 只需使用技能按请求完成任务。

**审查结果**：如果您无法打开浏览器（例如，Claude.ai的VM没有显示，或者您在远程服务器上），请完全跳过浏览器审查器。相反，直接在对话中呈现结果。对于每个测试用例，显示提示和输出。如果输出是用户需要查看的文件（如.docx或.xlsx），将其保存到文件系统并告诉他们在哪里，以便他们可以下载和检查它。内联询问反馈："这看起来如何？您会改变什么？"

**基准测试**：跳过定量基准测试 - 它依赖于基线比较，在没有子代理的情况下没有意义。专注于来自用户的定性反馈。

**迭代循环**：与以前相同 - 改进技能，重新运行测试用例，询问反馈 - 只是中间没有浏览器审查器。如果您有一个，您仍然可以将结果组织到文件系统上的迭代目录中。

**描述优化**：本节需要 `claude` CLI工具（特别是 `claude -p`），仅在Claude Code中可用。如果您在Claude.ai上，请跳过它。

**盲比较**：需要子代理。跳过它。

**打包**：`package_skill.py` 脚本在任何有Python和文件系统的地方都可以工作。在Claude.ai上，您可以运行它，用户可以下载生成的 `.skill` 文件。

**更新现有技能**：用户可能会要求您更新现有技能，而不是创建新技能。在这种情况下：
- **保留原始名称。** 注意技能的目录名称和 `name` 前言字段 - 保持不变。例如，如果安装的技能是 `research-helper`，输出 `research-helper.skill`（不是 `research-helper-v2`）。
- **在编辑前复制到可写位置。** 安装的技能路径可能是只读的。复制到 `/tmp/skill-name/`，在那里编辑，并从副本打包。
- **如果手动打包，首先在 `/tmp/` 中暂存**，然后复制到输出目录 - 直接写入可能因权限而失败。

## Cowork特定说明

如果您在Cowork中，主要要知道的是：

- 您有子代理，所以主要工作流程（并行生成测试用例、运行基线、评分等）都有效。（但是，如果您遇到严重的超时问题，可以串行运行测试提示而不是并行运行。）
- 您没有浏览器或显示，所以当生成评估查看器时，使用 `--static <output_path>` 写入独立的HTML文件而不是启动服务器。然后提供一个链接，用户可以点击在浏览器中打开HTML。
- 出于某种原因，Cowork设置似乎不太愿意Claude在运行测试后生成评估查看器，所以只是重申：无论您是在Cowork还是Claude Code中，在运行测试后，您都应该始终为人工生成评估查看器，以便在审查示例之前查看示例，使用 `generate_review.py`（而不是编写您自己的精品html代码）。提前道歉，但我要大写了：在评估输入之前生成评估查看器。您希望尽快将他们带到人工面前！
- 反馈工作方式不同：由于没有运行服务器，查看器的"提交所有评论"按钮将下载 `feedback.json` 作为文件。然后您可以从那里读取它（您可能必须先请求访问权限）。
- 打包有效 - `package_skill.py` 只需要Python和文件系统。
- 描述优化（`run_loop.py` / `run_eval.py`）在Cowork中应该可以正常工作，因为它使用 `claude -p` 通过子进程，而不是浏览器，但请保存它，直到您完全完成制作技能并且用户同意它处于良好状态。
- **更新现有技能**：用户可能会要求您更新现有技能，而不是创建新技能。遵循上面Claude.ai部分中的更新指导。
